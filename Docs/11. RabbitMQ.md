RabbitMQ is used for asynchronous messaging between services, implementing an Outbox pattern with Saga choreography for order processing. Here's how it works:

## Core Components

### 1. Configuration (RabbitMqOptions)
   • Connection URI (default: amqp://guest:guest@localhost:5672)

   • Main exchange: keycloak-demo.events (topic exchange)

   • Dead letter exchange: keycloak-demo.events.dlx for failed messages

   • Queues: payments.stub.requests, orders.saga, orders.execution.dispatch

   • Batch size for outbox processing: 20 messages

   • Retry delay: 5 seconds


### 2. Outbox Pattern (IntegrationEventOutbox<TContext>)

   • Messages are stored in the database (OutboxMessage table) before publishing

   • Ensures transactional consistency: business changes and message publishing are atomic

   • Messages include routing key, serialized payload, correlation ID, and timestamps

   • Supports at-least-once delivery guarantees


### 3. Outbox Dispatcher (OutboxDispatcher<TContext>)
   
• Background service that runs continuously in each service (Orders.Api, Payments.Api, etc.)

   • Fetches unpublished messages from the database in batches

   • Publishes to RabbitMQ with persistent delivery mode

   • Marks messages as published after successful send

   • Retries on failures with configurable delay

### 4. Topology Setup (RabbitMqTopology)
   • Declares topic exchanges (main and dead letter)

   • Creates queues bound to routing keys (e.g., orders.payment-requested, payments.*)
 
   • Sets up dead letter queues for failed message handling


### 5. Consumers
   • OrderSagaConsumer: Handles order lifecycle events (payment requested/authorized/failed, execution dispatched/started/completed/failed)

   • PaymentStubConsumer: Simulates payment processing

   • OrderExecutionDispatchConsumer: Dispatches order execution tasks

   • All consumers use manual acknowledgment (autoAck: false) for reliability

   • Implement idempotency via ConsumedMessages table to prevent duplicate processing

   • On failure, messages are negatively acknowledged and requeued


### 6. Message Flow for Order Processing

• Order created → OrderPaymentRequestedMessage published

• Payment service processes → PaymentAuthorizedMessage or PaymentFailedMessage

• Saga consumer updates order state and publishes OrderExecutionDispatchedMessage

• Tasks service receives and processes execution

• Final status updates flow back through the saga

### 7. Reliability Features
   • At-least-once delivery: No auto-ack, manual commit after processing

   • Idempotency: Checks ConsumedMessages to skip duplicates

   • Dead letter queues: Failed messages routed for manual inspection

   • Retries: Failed publishes/delivery retried with delays

   • Correlation IDs: Trace messages across services

   • Versioned messages: Include version field for schema evolution

### 8. Integration
   • Services register RabbitMqConnectionFactory and OutboxDispatcher<TContext> as hosted services

   • Connection string provided via Aspire AppHost configuration

   • Messages serialized/deserialized using JSON with type information
   This implementation follows distributed systems best practices, ensuring reliable message delivery and eventual consistency across your microservices architecture.
