Detailed architectural improvements (modern approaches)

### A. Turn “MediatR usage” into a real vertical-slice/CQRS setup

Right now you already have a good start: feature folders for **Requests** and **Handlers** (Orders/Tasks/Users). ([GitHub][1])
To make it feel “modern production” (and interview-ready), push it further:

**1) Make controllers ultra-thin**
Today controllers still do:

* input validation (`if (req.TotalAmount <= 0) ...`)
* user resolution + impersonation logic (`asUserId`, admin check)
* sometimes manual branching (“digital vs physical”) ([GitHub][2])

**Better:**

* Controllers only map HTTP → MediatR request and return result.
* Move *all* validation to pipeline (FluentValidation) or to request validators.
* Move “effective user” resolution into a service (see next section), then handlers depend on `ICurrentUser`.

This is the “vertical slice” feel: controller becomes transport-only.

**2) Standardize request shapes**
Make a consistent split:

* `Query : IRequest<TResponse>`
* `Command : IRequest<Result<TResponse>>` (or `OneOf<T, NotFound, ValidationError>`)

You already return mixed results in places (`UpdateOrderCommand` returns `{ NotFound, ValidationError, Order }`). ([GitHub][2])
That’s good, but I’d unify it across all handlers so the API behavior is consistent.

---

### B. Extract “current user” + impersonation into a dedicated layer

You currently duplicate these patterns in controllers:

* “Get or create current user” based on `sub` claim ([GitHub][3])
* Impersonation via `asUserId` guarded by role `"admin"` ([GitHub][3])

**Problems (interview topics):**

* It’s repeated across controllers.
* It ties controllers to EF (`AppDbContext` injected into controller methods).
* It’s easy to accidentally forget checks in a new controller.

**Better: introduce**

* `ICurrentUserAccessor` (reads claims)
* `IEffectiveUserResolver` (handles `asUserId` + admin policy + loads user)
* Optional: `IImpersonationPolicy` (central policy rules)

Then handlers can do:

* `var userId = _effectiveUser.UserId;`
  instead of controllers passing `user.Id` into every request.

If you like “cleaner API”, you can also remove `asUserId` from public endpoints and expose impersonation only under `/admin/*` routes.

---

### C. Replace ad-hoc validation with MediatR pipeline behaviors

Your controllers do manual validation checks (title required, amount > 0, etc.). ([GitHub][2])

**Modern approach:**

* Add `FluentValidation` validators per request DTO / command.
* Add MediatR pipeline behavior: `ValidationBehavior<TRequest,TResponse>`.

Benefits:

* Controllers stay thin.
* Validation rules live next to the request, not in transport layer.
* Interviewers love this because it demonstrates clean separation + cross-cutting concerns.

---

### D. Add cross-cutting behaviors: logging, performance, transactions

Right now your auth events log with `Console.WriteLine`. ([GitHub][4])
In production, you’ll want consistent structured logging.

**Add pipeline behaviors:**

1. `LoggingBehavior` — log request name + key fields (no secrets).
2. `PerformanceBehavior` — warn if > X ms.
3. `TransactionBehavior` — wrap command handlers in a DB transaction (when needed), especially if you’ll add outbox/events later.

Then replace `Console.WriteLine` in JWT events with `ILogger` (structured logs).

---

### E. Improve Keycloak role mapping: move it to ClaimsTransformation or config

You’re currently parsing `realm_access.roles` in `OnTokenValidated` and adding `ClaimTypes.Role` claims manually. ([GitHub][4])

This works, but a cleaner setup is:

* Implement `IClaimsTransformation` to map Keycloak realm/client roles → `ClaimTypes.Role`.
* Keep JWT events minimal (debug only).

It also becomes testable and reusable across services.

---

### F. Data access improvements (EF Core)

Your handlers generally use `AsNoTracking()` and map to DTOs. That’s good. ([GitHub][5])
Next steps to look more “senior”:

* Use **projection** (`Select(new Dto(...))`) so the DB returns only needed columns (instead of loading entities then mapping).
* Add **pagination** for list endpoints (`GET /tasks`, `GET /orders`).
* Consider **specification-ish** approach only if queries get complex; otherwise keep it simple.

---

### G. Error handling: stop returning ad-hoc strings; use ProblemDetails

Right now you return `BadRequest("...")`, `NotFound("...")`, etc. ([GitHub][2])
That’s fine for demos, but modern APIs usually standardize errors:

* Add global exception middleware
* Use RFC7807 `ProblemDetails` (built-in in ASP.NET Core)
* Map domain/application errors → consistent HTTP responses

This also makes the frontend easier.

---

### H. Project structure: one more step toward “Clean Architecture” (without overdoing it)

Your backend currently appears as a single project (`backend.csproj`) with folders. ([GitHub][6])
For interview-ready modernization, a common sweet spot is 3 projects:

* `Api` (controllers, auth, DI)
* `Application` (MediatR requests/handlers, behaviors, validators, DTOs)
* `Infrastructure` (EF DbContext, migrations, Keycloak integration, external services)

This keeps MediatR + business logic out of the API layer and makes testing easier.

# MediatR Implementation Guide

## Overview
MediatR is implemented as a **CQRS (Command Query Responsibility Segregation)** library for handling requests and responses in a decoupled, pipeline-based manner. It enables in-process messaging within each service/API, allowing controllers to dispatch commands and queries without directly depending on their handlers.

---

## 1. Core Concepts and Interfaces

### Requests
Commands and queries inherit from MediatR's `IRequest<TResponse>`. The project defines custom abstractions:

| Interface | Purpose | Example |
|-----------|---------|---------|
| `ICommand<TResponse>` | Write operations (creating/updating data) | `CreateOrderCommand` |
| `IQuery<TResponse>` | Read operations (fetching data) | `GetOrdersQuery` |

> These are simple **marker interfaces** that extend `IRequest<TResponse>`, ensuring type safety for responses.

### Handlers
Implement `IRequestHandler<TRequest, TResponse>` to process requests:

- Each handler is responsible for a **single request type** and returns a response
- Handlers can inject dependencies (e.g., repositories, services) via DI
- Complex handlers can dispatch other requests using `IMediator.Send()` for chaining

**Example:** `CreateOrderHandler` dispatches `CreateDigitalOrderCommand` or `CreatePhysicalOrderCommand` based on input.

### Dispatching
Controllers use `ISender` (a MediatR interface) to send requests:

```csharp
var result = await _sender.Send(request, cancellationToken);
```

---

## 2. Registration and Setup

### Service Registration
MediatR is registered in each API's `Program.cs` using:

```csharp
builder.Services.AddMediatR(cfg => 
    cfg.RegisterServicesFromAssembly(typeof(Program).Assembly));
```

| API Type | Assembly Registered | Behaviors |
|----------|---------------------|-----------|
| **Feature APIs** (e.g., `backend.Orders.Api`) | Respective request assembly (e.g., `backend.Orders.Requests.Orders.CreateDigitalOrderCommand.Assembly`) | None explicitly registered |
| **Main API** (`backend.Api`, gateway) | Program assembly | Behaviors registered (see §4) |
| **Other APIs** (Tasks, Users, etc.) | Respective request assemblies | Follow similar pattern |

### Dependency Injection
- Handlers, validators, and behaviors are resolved via **DI**
- Requests are **discovered automatically** by MediatR—no explicit registration needed

---

## 3. Usage in Controllers

Controllers inject `ISender` (not `IMediator`, for simplicity) and use it to dispatch requests.

### Example: `OrdersController`

```csharp
// Query example
var result = await _sender.Send(new GetOrdersQuery(), ct);

// Command example
var result = await _sender.Send(command, ct);
```

### Response Pattern
Responses are typically wrapped in `Result<T>` (e.g., `Result<OrderViewDto>`), which includes:
- Success/failure status
- Errors for consistent API responses

### Controller Responsibilities
```csharp
// Return success
return Ok(result.Value);

// Return validation errors
return BadRequest(result.Errors);
```

> **Business logic is isolated in handlers**—controllers handle only HTTP concerns.

---

## 4. Pipeline Behaviors

Behaviors implement `IPipelineBehavior<TRequest, TResponse>` to add **cross-cutting concerns** around handler execution.

### Registered in `backend.Api\Program.cs`
Registered as transients, applying to all requests:

| Behavior | Purpose | Target |
|----------|---------|--------|
| `ValidationBehavior<TRequest, TResponse>` | Runs FluentValidation on commands | Commands only (`ICommand<>`) |
| `LoggingBehavior<TRequest, TResponse>` | Logs request details (type, duration) | All requests |
| `PerformanceBehavior<TRequest, TResponse>` | Monitors execution time, logs slow requests | All requests |
| `TransactionBehavior<TRequest, TResponse>` | Wraps handlers in DB transactions (EF Core) | Commands |

### Pipeline Flow
```csharp
public async Task<TResponse> Handle(
    TRequest request, 
    RequestHandlerDelegate<TResponse> next, 
    CancellationToken cancellationToken)
{
    // Pre-processing (e.g., validation)
    
    var response = await next(); // Proceed to next behavior or handler
    
    // Post-processing (e.g., logging results)
    
    return response;
}
```

> **Note:** In feature APIs without explicit behavior registration, handlers run directly without these enhancements—keeping them lightweight.

---

## 5. Request Flow Example

**Scenario:** `POST /api/orders` in `OrdersController`

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   Controller    │────▶│    MediatR      │────▶│   Behaviors     │
│  (receives DTO) │     │   (routes req)  │     │ (if registered) │
└─────────────────┘     └─────────────────┘     └────────┬────────┘
                                                         │
                              ┌────────────────────────┘
                              ▼
                    ┌─────────────────┐
                    │     Handler     │
                    │ (business logic)│
                    │  - Validates    │
                    │  - Dispatches   │
                    │    sub-commands │
                    └────────┬────────┘
                              │
                    ┌─────────┘
                    ▼
           ┌─────────────────┐
           │   Response      │
           │  bubbles back   │
           │  through layers │
           └─────────────────┘
```

**Step-by-step:**

| Step | Component | Action |
|------|-----------|--------|
| i | `OrdersController` | Receives `CreateOrderCommand`, calls `_sender.Send(command, ct)` |
| ii | MediatR | Routes to `CreateOrderHandler` based on registered assemblies |
| iii | Pipeline Behaviors | Validation runs first; if valid, logging/performance/transaction wrap the handler |
| iv | Handler | Processes request (switches on `OrderType`, dispatches sub-commands via `IMediator`) |
| v | Response | Bubbles back through behaviors to controller, which returns HTTP result |

---

## 6. Benefits and Patterns

| Pattern | Implementation |
|---------|----------------|
| **Decoupling** | Controllers don't know about handlers or business logic |
| **CQRS** | Commands mutate state; queries read it, with separate handlers |
| **Modular Setup** | Each service (Orders, Tasks, Users, Payments) has its own handlers and DB context |
| **Async/Await** | All handlers and behaviors use `CancellationToken` and async patterns |
| **Extensibility** | Easy to add new behaviors (caching, circuit breakers) or handlers without changing controllers |

> **Best for:** Microservices-style monoliths, keeping each feature isolated while allowing complex request chaining within services.
