The project is **already halfway between a modular monolith and microservices** rather than being a classic single monolith. The backend folder contains a main `backend.Api`, a separate `backend.Auth.Api`, an Aspire `backend.AppHost`, and feature projects for `Users`, `Orders`, `Tasks`, and `Payments`. The main API wires MediatR handlers from multiple feature assemblies and exposes separate controllers for users, orders, and tasks. ([GitHub][1])

The strongest signal that it is **not yet real microservices** is the data layer. The shared `AppDbContext` still owns tables/sets for tasks, task comments, orders, saga state, outbox messages, consumed messages, and payment event records in one EF Core context, which means those domains still share one persistence boundary and one transaction boundary. That is a modular monolith pattern, not service autonomy. ([GitHub][2])

There is one meaningful exception already present: `backend.Auth.Api` is a separate service with its own `Auth` connection string and its own internal users controller, while the main API configures an `AuthService` HTTP client. That suggests user-directory/auth-profile concerns have already started to move out of the main app boundary. ([GitHub][3])

Also, the repo already uses an Aspire AppHost to orchestrate multiple projects and shared infrastructure. In this repo, `backend.AppHost` starts at least `api` and `auth-api`, injects separate connection strings, and wires RabbitMQ. That fits Aspire’s role as an orchestration layer for distributed apps. Microsoft’s Aspire guidance also describes AppHost as the orchestration project and ServiceDefaults as shared configuration defaults. ([GitHub][4])

## Recommendation

Do **not** do a big-bang rewrite into many services at once.

This codebase is in a good place for an **incremental strangler-style refactor**:

1. Keep the current modular structure.
2. Promote the existing feature modules into independently deployable services one by one.
3. Split databases only when a boundary is proven.
4. Preserve RabbitMQ + outbox for cross-service events.

That approach aligns with standard microservice guidance: each microservice should own its domain logic and private data, and cross-service consistency should rely on async messaging/outbox rather than distributed transactions. ([Microsoft Learn][5])

## What I would refactor into services

### 1. Auth/User Directory service

This one is the easiest because it already exists in embryonic form as `backend.Auth.Api`. It has its own DB connection and internal user endpoints. The main API should stop directly thinking of “users” as local CRUD entities and treat auth-profile data as owned by this service. ([GitHub][6])

Target responsibility:

* local projection of Keycloak identities
* ensure/find/create/update user profiles
* publish `UserCreated`, `UserUpdated`, `UserDeleted` events
* serve internal lookups for other services, or better, provide replicated read models via events

### 2. Orders service

Orders are already a distinct module with dedicated requests, handlers, mappers, validation, and controller endpoints. The presence of order saga state and payment timeline/workflow endpoints makes Orders a strong domain boundary. ([GitHub][7])

Target responsibility:

* order aggregate
* order lifecycle/workflow
* order read endpoints
* initiating payment workflow
* owning `orders` + `order_saga_states` data

### 3. Tasks service

Tasks are a separate feature module with their own controller and comment subresource. They do not appear tightly coupled to orders except shared infrastructure and DB. That makes them another good extraction candidate. ([GitHub][8])

Target responsibility:

* task aggregate
* task comments
* task list/read/update/delete
* optional projection of user display info, but not user ownership data itself

### 4. Payments service

Right now payments look like infrastructure plus event records, not a full standalone API surface yet. Because order-payment workflow already exists, I would separate Payments after Orders, not before. ([GitHub][9])

Target responsibility:

* payment attempts/statuses
* external provider integration
* idempotency keys
* payment event history
* publish `PaymentSucceeded` / `PaymentFailed`

## Best target architecture

A pragmatic end-state would be:

* `api-gateway` or BFF for frontend
* `auth-service`
* `orders-service`
* `tasks-service`
* `payments-service`
* RabbitMQ
* one database per service
* shared observability via Aspire `ServiceDefaults`

That maps well to the repo’s current direction because the AppHost is already orchestrating multiple apps/resources, and the current shared modules can be reduced over time to only reusable technical packages. ([GitHub][4])

## What should stay shared vs what must stop being shared

### Keep shared

Good candidates for shared libraries:

* contracts/events DTOs
* common result/error primitives
* telemetry/logging helpers
* auth token parsing helpers
* maybe MediatR behavior abstractions if still useful inside each service

### Stop sharing

These should become service-private:

* EF Core `DbContext`
* entity models
* repositories
* migrations
* service-specific validators/handlers
* transaction behaviors tied to one database

Right now the shared `AppDbContext` is the biggest blocker because it enables cross-domain transactions inside one process. In real microservices, each service owns its DB, and transactions remain local to that service. ([GitHub][2])

## Concrete migration plan

### Phase 0: stabilize the modular monolith

Before extracting anything:

* keep `backend.Api` as the frontend-facing entrypoint
* make each feature module expose only application contracts
* ban direct feature-to-feature EF usage
* introduce explicit integration events for cross-module communication
* separate read models from write models where needed

This reduces accidental coupling before network boundaries are introduced.

### Phase 1: finish extracting Auth

You already have:

* separate `backend.Auth.Api`
* separate `Auth` connection string
* internal user endpoints
* an HTTP client from main API to auth service ([GitHub][3])

Next steps:

* move all user CRUD ownership into auth service
* main API becomes a client/consumer only
* publish user events so Orders/Tasks can maintain local projections if needed
* remove shared `AppUser` persistence from non-auth services

### Phase 2: extract Tasks service

This is likely the cleanest first business-domain split because tasks appear operationally independent. The current `TasksController` can be moved almost as-is into a separate service boundary. ([GitHub][10])

Do this:

* create `backend.Tasks.Api`
* create `TasksDbContext`
* move task entities/migrations into tasks service
* create task-specific outbox table
* route `/api/tasks` through gateway/BFF or keep reverse proxy routing

### Phase 3: extract Orders service

Orders have more coupling because of payment workflow and saga state, but they are also a stronger business boundary. ([GitHub][11])

Do this:

* create `backend.Orders.Api`
* move `orders`, `order_saga_states`, and order outbox into Orders DB
* emit events:

  * `OrderCreated`
  * `OrderPaymentRequested`
  * `OrderCancelled`
  * `OrderCompleted`
* stop direct joins/calls into shared DB tables

### Phase 4: extract Payments service

Once Orders emits payment-requested events, Payments can become a true autonomous processor. The current payment records can move into its own database and be consumed asynchronously. ([GitHub][2])

### Phase 5: remove the old aggregate API

At that point `backend.Api` can become:

* API gateway/BFF only, or
* be retired if frontend calls services through a gateway/reverse proxy

## Messaging strategy

RabbitMQ is already present, and the code already includes RabbitMQ wiring plus outbox-related tables. That is exactly the right direction for microservice extraction, because once databases are split you should avoid distributed transactions and use local transactions + outbox + idempotent consumers. ([GitHub][3])

I would standardize on:

* command via HTTP/gRPC only when immediate response is needed
* events via RabbitMQ for state propagation
* one outbox table per service DB
* one inbox/consumed-messages table per service for idempotency

That is much safer than trying to coordinate multi-service writes with 2PC; Microsoft’s microservices guidance explicitly points to async communication and outbox-style approaches instead of distributed transaction coordination for modern systems. ([Microsoft Learn][12])

## Biggest risks in this repo if you extract too early

1. **Shared database coupling**
   The current single `AppDbContext` means services likely assume same-transaction consistency. Splitting that too fast will break workflows. ([GitHub][2])

2. **Over-shared domain model**
   If `backend.Domain` stays a shared entity package, you will get distributed monoliths instead of microservices. The repo currently centralizes models and EF config in `backend.Domain`. ([GitHub][13])

3. **Synchronous auth dependency**
   Main API currently calls auth service over HTTP. If that remains on the hot path for every request, latency and failure coupling will increase. Prefer token claims for identity and local projections for read-heavy user data. ([GitHub][3])

4. **Gateway confusion**
   If the frontend keeps hitting many internal services directly, auth/cors/versioning get messy. A gateway or BFF will keep the frontend stable during migration.

## Bottom line

My conclusion after reviewing the repo:

* **You should refactor it into microservices incrementally, not rewrite it.**
* **The current code is best described as a modular monolith plus one extracted auth service.**
* **The best first real extraction is Auth, then Tasks, then Orders, then Payments.**
* **The key technical move is replacing the shared `AppDbContext` with database-per-service boundaries while leaning on RabbitMQ + outbox for consistency.** ([GitHub][3])

## Example:

```csharp
var builder = DistributedApplication.CreateBuilder(args);

var rabbit = builder.AddRabbitMQ("messaging");

var authDb = builder.AddPostgres("auth-db");
var tasksDb = builder.AddPostgres("tasks-db");
var ordersDb = builder.AddPostgres("orders-db");

builder.AddProject<Projects.Auth_Api>("auth")
    .WithReference(authDb)
    .WithReference(rabbit);

    .WithReference(tasksDb)
    .WithReference(rabbit);

builder.AddProject<Projects.Orders_Api>("orders")
    .WithReference(ordersDb)
    .WithReference(rabbit);

builder.Build().Run();
```
