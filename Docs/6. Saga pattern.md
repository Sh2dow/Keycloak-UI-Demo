Moving to Saga Pattern


• We should move to Saga Pattern, but only for the cross-service order and payment workflow, not as a blanket pattern across the whole system.

  For this repo, a Saga becomes justified once these are truly separate services:

  - Orders
  - Payments
  - later possibly Inventory / Fulfillment / Notifications

  Why it fits here:

  - order execution now depends on asynchronous payment outcome
  - you no longer have a single DB transaction across services
  - failures need compensation or terminal states instead of rollback

  Where to use it:

  - Create Order -> Request Payment -> Payment Authorized/Failed -> Dispatch Order Execution
  - later: Dispatch -> Fulfillment Started -> Fulfillment Failed -> Compensate / mark manual review

  Where not to use it:

  - inside a single service boundary
  - for simple CRUD flows
  - for Tasks right now
  - for local EF Core transactions already handled in one module

  Recommended approach:

  - start with an orchestration-style saga for Orders and Payments
  - let Order Service own the saga state initially
  - persist saga state explicitly in the orders database
  - drive transitions from RabbitMQ events
  - make every consumer idempotent
  - model compensation as state transitions, not distributed rollback

  Suggested first saga states:

  - Created
  - PaymentPending
  - PaymentAuthorized
  - PaymentFailed
  - ExecutionDispatched
  - Completed
  - ManualReview

  Suggested events:

  - OrderCreated
  - OrderPaymentRequested
  - PaymentInitiated
  - PaymentAuthorized
  - PaymentFailed
  - OrderExecutionDispatched
  - OrderExecutionFailed
  - OrderCompleted

  My recommendation:

  - do move toward Saga Pattern for the order-payment-execution flow
  - do not introduce a generic saga framework yet
  - implement one concrete saga first in the Orders module and prove the model
  - keep it small, explicit, and code-driven before abstracting
