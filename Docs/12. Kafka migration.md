### RabbitMQ Event Sourcing Characteristics

| Aspect | Behavior | Limitation |
|--------|----------|------------|
| **Event Origin** | Outbox table → Dispatcher → Exchange → Queue | Single point of dispatch |
| **Retention** | Queue-level TTL or dead-letter | Messages deleted after ACK |
| **Replay** | ❌ Not possible | Once consumed, event is gone |
| **Projection Source** | Live consumption only | Can't rebuild from history |
| **Fan-out** | Multiple queues per exchange | Each message copied per queue |

### Kafka Event Sourcing Characteristics

| Aspect | Behavior | Advantage |
|--------|----------|-----------|
| **Event Origin** | Outbox → CDC (Debezium) OR direct producer | Dual path support |
| **Retention** | Disk-persisted, time/size-based | Days/weeks/forever |
| **Replay** | ✅ Seek to any offset | Rebuild projections from t=0 |
| **Projection Source** | Immutable event log | Same events, multiple views |
| **Fan-out** | Consumer groups | Each group gets all events |

| **Timeline/Audit** | Not possible (events gone) | Replay from offset 0 | Redis / JSONB |

| **Search Index** | Manual indexing | Stream to Elastic | Elasticsearch |

## 7. Key Decision Matrix

| Requirement | RabbitMQ | Kafka |
|-------------|----------|-------|
| Simple messaging | ✅ Excellent | ⚠️ Overkill |
| Event replay / rebuild projections | ❌ Impossible | ✅ Native |
| Multiple independent projections | ❌ Complex | ✅ Consumer groups |
| High throughput (>100K msg/s) | ⚠️ Limited | ✅ Designed for it |
| Message retention (auditing) | ❌ Ephemeral | ✅ Configurable |
| Stream processing (real-time analytics) | ❌ Not supported | ✅ Kafka Streams |
| Operational complexity | ✅ Simple | ⚠️ Complex |
| Schema evolution | ⚠️ Manual | ✅ Schema Registry |

## Summary

**RabbitMQ** works as a message broker for your current saga choreography, but **Kafka unlocks true event sourcing** where the event log becomes the source of truth for multiple specialized read models. The critical difference is **replayability**: with Kafka, projections are derived views that can be rebuilt at any time by replaying the immutable event log, while RabbitMQ forces you to treat consumed messages as ephemeral triggers with no historical record.
