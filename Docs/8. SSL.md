# SSL / HTTPS Plan

## Goal

Expose Keycloak and the application over HTTPS for real external traffic.

Current state:
- EC2 exposes `8080`, `5000`, `5001`, `5173` directly
- Keycloak runs with HTTP enabled
- external access to Keycloak hits `http://<ip>:8080`
- Keycloak realm policy still requires SSL for external requests

Target state:
- public traffic terminates TLS at an AWS Application Load Balancer
- ALB forwards traffic to EC2 over HTTP on private target ports
- Keycloak trusts forwarded headers and knows its public HTTPS hostname
- direct public access to Keycloak/app ports is removed

## Recommended Architecture

Public endpoints:
- `https://auth.<your-domain>` -> Keycloak
- `https://app.<your-domain>` -> frontend
- optional: `https://api.<your-domain>` -> backend API

AWS resources:
- ACM certificate for the public hostnames
- one Application Load Balancer
- one HTTPS listener on `443`
- host-based routing rules
- target groups:
  - Keycloak -> EC2 port `8080`
  - Frontend -> EC2 port `5173`
  - Backend API -> EC2 port `5000` if exposed separately

Traffic flow:
- client -> ALB over HTTPS
- ALB -> EC2 over HTTP
- Keycloak/backend read `X-Forwarded-*` headers from ALB

## Keycloak Configuration

For edge TLS termination, Keycloak should:
- keep HTTP enabled internally
- use a fixed public hostname
- trust proxy headers

Required settings:
- `KC_HTTP_ENABLED=true`
- `KC_PROXY_HEADERS=xforwarded`
- `KC_HOSTNAME=https://auth.<your-domain>`

Recommended:
- `KC_HOSTNAME_STRICT=true`

Current compose should be adjusted from permissive dev mode to explicit hostname mode.

## Backend Configuration

Backend auth settings should use the external HTTPS Keycloak authority:
- `Keycloak__Authority=https://auth.<your-domain>/realms/myrealm`

Metadata can usually use the same public URL once ALB is in place:
- `Keycloak__MetadataAddress=https://auth.<your-domain>/realms/myrealm/.well-known/openid-configuration`

Do not keep public IP based Keycloak authority in production.

## Frontend Configuration

Frontend should point OIDC to the HTTPS hostname:
- `VITE_KEYCLOAK_AUTHORITY=https://auth.<your-domain>/realms/myrealm`

If frontend is also behind ALB, prefer:
- `https://app.<your-domain>`

## AWS Steps

### 1. Request ACM certificate

Create an ACM certificate in the same region as the ALB.

Example hostnames:
- `auth.<your-domain>`
- `app.<your-domain>`
- `api.<your-domain>` if needed

Validate by DNS.

### 2. Create ALB

Create:
- internet-facing ALB
- in public subnets
- security group allowing:
  - inbound `443` from `0.0.0.0/0`
  - outbound to EC2 target ports

### 3. Create target groups

Suggested target groups:
- `keycloak-tg` -> port `8080`
- `frontend-tg` -> port `5173`
- `backend-api-tg` -> port `5000`

Register the EC2 instance as the target.

### 4. Create HTTPS listener

Listener:
- port `443`
- ACM certificate attached

Routing:
- `Host: auth.<your-domain>` -> `keycloak-tg`
- `Host: app.<your-domain>` -> `frontend-tg`
- `Host: api.<your-domain>` -> `backend-api-tg`

### 5. Update DNS

Create DNS records:
- `auth.<your-domain>` -> ALB DNS name
- `app.<your-domain>` -> ALB DNS name
- `api.<your-domain>` -> ALB DNS name if used

### 6. Lock down EC2 security group

After ALB works:
- remove public ingress for:
  - `8080`
  - `5000`
  - `5001`
  - `5173`
- allow those ports only from the ALB security group

Keep `22` only if SSH is still needed.

## Compose / Runtime Changes Needed

### Keycloak

Target environment values:

```yaml
KC_HTTP_ENABLED: "true"
KC_PROXY_HEADERS: xforwarded
KC_HOSTNAME: https://auth.<your-domain>
KC_HOSTNAME_STRICT: "true"
```

### Backend API

Target environment values:

```yaml
Keycloak__Authority: https://auth.<your-domain>/realms/myrealm
Keycloak__MetadataAddress: https://auth.<your-domain>/realms/myrealm/.well-known/openid-configuration
```

### Frontend build arg

Target:

```yaml
VITE_KEYCLOAK_AUTHORITY=https://auth.<your-domain>/realms/myrealm
```

## Realm SSL Policy

Once HTTPS is live through ALB:
- keep Keycloak realm SSL required

Do not set realm SSL to `NONE` in production.

## Security Notes

- Do not expose Keycloak directly on `8080` to the public internet in production.
- Do not expose backend API directly if ALB is the intended ingress layer.
- Keep RDS private.
- Prefer ALB security group -> EC2 security group rules instead of public CIDRs.

## Rollout Order

1. Provision ACM certificate
2. Provision ALB and target groups
3. Point DNS to ALB
4. Update Keycloak hostname/proxy settings
5. Update backend/frontend OIDC URLs to HTTPS hostnames
6. Verify login flow
7. Close direct EC2 public ports

## Verification Checklist

- `https://auth.<your-domain>/realms/master` responds
- Keycloak login page opens without `HTTPS required`
- frontend login redirect goes to HTTPS hostname
- backend accepts JWT tokens issued with HTTPS issuer
- ALB target groups are healthy
- EC2 ports `8080`, `5000`, `5001`, `5173` are no longer public
