# Moving to Microservices Architecture

Move to microservices only through an incremental strangler approach. This codebase is a good candidate for eventual service split because the business domains are already separated by folders and handlers (`Users`, `Orders`, `Tasks`), but it is not ready for a clean cut today because it still shares one process, one DI container, one EF Core context, one database schema, and one transaction boundary across all commands in `Program.cs:21`, `AppDbContext.cs:6`, and `TransactionBehavior.cs:7`.

The biggest blocker is that `Users` is currently the identity source for the whole app and is coupled into runtime request handling through `EffectiveUserAccessor.cs:8`. There is also DTO leakage from `Orders` into `Users`; for example, `UserWithOrdersDto` includes order data directly in `UserDtos.cs:7`. That means the correct near-term goal is not "break the monolith apart now"; it is "turn the current app into a modular monolith with explicit contracts, then extract services one by one".

## Target Architecture

Recommended target services:

- `Identity/User Service`
  Owns user profile, local user provisioning from Keycloak subject, impersonation rules, and user read model.
- `Order Service`
  Owns order lifecycle, digital and physical order variants, order persistence, and order status transitions.
- `Payment Service`
  Starts as a stub, owns payment workflow orchestration, payment event stream, and dispatching order execution after payment outcome is known.
- `Task Service`
  Owns task items, task comments, task status and priority rules, and task queries.
- `API Gateway/BFF`
  Handles routing, auth enforcement, rate limiting, and response aggregation for the frontend.
- `Shared platform services`
  Keycloak, observability stack, message broker, service discovery/config, and possibly centralized audit logging.

Recommended ownership rules:

- Each service owns its own database schema or database.
- No direct EF navigation properties across service boundaries.
- Cross-service communication uses HTTP/gRPC for queries and RabbitMQ events for state propagation.
- Authentication stays centralized in Keycloak; authorization policy can be partly centralized and partly service-local.
- Payment flow should be event-driven from the start so order execution is dispatched from payment outcome events rather than synchronous order-to-payment calls.

## Migration Plan

### Phase 0: Stabilize the monolith

Duration: 1 to 2 weeks.

Work:

- Add architecture tests or conventions so `Users`, `Orders`, and `Tasks` stop referencing each other directly except through explicit contracts.
- Introduce structured logging, correlation IDs, OpenTelemetry tracing, health checks, and metrics.
- Baseline current latency, error rate, deployment time, and DB query performance.
- Document current APIs, ownership boundaries, and transactional assumptions.

Exit criteria:

- You can measure whether the migration is improving or harming the system.
- Teams know which code belongs to which future service.

### Phase 1: Convert to a modular monolith

Duration: 2 to 4 weeks.

Work:

- Split the solution into projects such as:
  - `Backend.Api`
  - `Backend.Users`
  - `Backend.Orders`
  - `Backend.Tasks`
  - `Backend.SharedKernel` or, preferably, very small shared abstractions only
- Replace folder-based separation with assembly/project boundaries.
- Introduce per-module repositories or query services so modules do not access unrelated tables.
- Remove DTO leakage like `UserWithOrdersDto -> OrderViewDto`; use composition at the API layer instead.
- Move all domain rules out of controllers into application/domain modules.

Exit criteria:

- The monolith still deploys as one app, but each business area is isolated enough to extract later.
- No module depends on another module's EF entities.

### Phase 2: Remove shared database assumptions

Duration: 2 to 4 weeks.

Work:

- Split `AppDbContext` into module-owned contexts first, even if they still point to the same PostgreSQL instance.
- Remove navigation properties crossing domain boundaries:
  - `AppUser -> Orders`
  - `AppUser -> Tasks`
  - `TaskItem -> User`
- Replace cross-module foreign-key assumptions with stable external IDs (`UserId`, `OrderId`, and so on) plus service-level validation.
- Remove global transaction assumptions from `TransactionBehavior.cs:7`; transactions should become module-local only.

Exit criteria:

- Each module can operate without ORM joins to another module's data.
- Breaking the DB apart later no longer changes business behavior.

### Phase 3: Establish integration patterns

Duration: 2 to 3 weeks.

Work:

- Introduce RabbitMQ as the system message broker and standard event bus.
- Start publishing domain events:
  - `UserProvisioned`
  - `UserUpdated`
  - `OrderCreated`
  - `OrderPaymentRequested`
  - `OrderStatusChanged`
  - `PaymentInitiated`
  - `PaymentAuthorized`
  - `PaymentFailed`
  - `OrderExecutionDispatched`
  - `TaskCreated`
  - `TaskCommentAdded`
- Use the outbox pattern in the monolith before extraction.
- Design the payment flow as event sourced from the beginning, even while the first version of `Payment Service` remains a stub.
- Build read models where needed instead of synchronous joins.
- Define service contracts using OpenAPI or protobuf.

Exit criteria:

- Cross-domain data dependencies are fulfilled by APIs or events, not shared tables.

### Phase 4: Extract the first service

Duration: 3 to 5 weeks.

Extract `Task Service` first.

Reason:

- It is the most self-contained domain.
- Its main dependency is the effective user identity.
- It has fewer cross-domain read requirements than `Users` and less variant complexity than `Orders`.

Work:

- Create a standalone `Task Service` with its own DB.
- Keep Keycloak JWT validation in the service, or validate at the gateway and forward identity claims securely.
- Resolve user ownership through `sub` claim or internal `UserId` mapping API.
- Route `/api/tasks` through the gateway to the new service.
- Keep the rest in the monolith.

Exit criteria:

- Tasks run independently in production.
- The frontend sees no contract break except possibly base URL or gateway routing.

### Phase 5: Extract `Payment Service` stub and RabbitMQ flow

Duration: 3 to 5 weeks.

Work:

- Create a standalone `Payment Service` stub with its own database and event store.
- Introduce RabbitMQ exchanges, queues, retry policies, dead-letter queues, and message versioning conventions.
- Emit `OrderPaymentRequested` from `Order Service` when an order reaches the payment-required state.
- Let `Payment Service` consume payment requests, append payment events to its stream, and publish outcome events such as `PaymentAuthorized` or `PaymentFailed`.
- Dispatch order execution from payment outcome events by publishing `OrderExecutionDispatched` only after the payment stream reaches an executable state.
- Keep external payment gateway integration stubbed initially so the event contracts and dispatching flow are validated before adding real provider logic.

Exit criteria:

- RabbitMQ is the default asynchronous integration path.
- Payment workflow runs through events end-to-end, even if provider calls are stubbed.
- Orders can react to payment outcome without synchronous coupling to payment logic.

### Phase 6: Extract `Order Service`

Duration: 3 to 6 weeks.

Work:

- Move order creation, update, and query logic into a standalone service.
- Give it a dedicated DB and migration pipeline.
- Publish order events for downstream payment and reporting flows.
- Build a lightweight order summary read model for the user-facing composite API if needed.
- Make order execution idempotent because `PaymentAuthorized` and dispatch events may be retried by RabbitMQ consumers.

Exit criteria:

- Orders are no longer stored in the monolith DB.
- No synchronous joins from users to orders remain.
- Order execution is triggered from payment events rather than in-process order handlers.

### Phase 7: Refactor `Users` into an identity/profile service

Duration: 3 to 6 weeks.

Work:

- Separate identity from Keycloak from the local application profile.
- Move current logic from `EffectiveUserAccessor.cs:8` into a dedicated user/profile service or shared identity gateway component.
- Decide whether local users are provisioned:
  - lazily on first request, or
  - asynchronously on `user-created` events from Keycloak/admin workflows.
- Move impersonation policy into a centralized access service or gateway policy.

Exit criteria:

- The monolith no longer owns the canonical user model.
- Other services use user/profile APIs or replicated read models.

### Phase 8: Introduce gateway/BFF composition

Duration: 2 to 3 weeks.

Work:

- Add an API gateway such as YARP, Ocelot, or an ingress plus dedicated BFF.
- Keep frontend-facing contracts stable while internal services evolve.
- Compose responses such as user with orders in the gateway/BFF instead of inside the user domain.

Exit criteria:

- Clients call one entry point.
- Internal service topology can change without breaking the UI.

### Phase 9: Decommission the monolith

Duration: 1 to 2 weeks.

Work:

- Remove leftover endpoints from the monolith.
- Shut down shared DB access.
- Archive old migrations and ownership docs.
- Finalize service SLOs and on-call ownership.

## Recommended Service Boundaries for This Repo

- `Users`
  Owns `AppUser`, user profile, impersonation rules, and Keycloak subject mapping.
- `Orders`
  Owns `Order`, `DigitalOrder`, `PhysicalOrder`, order status rules, and emits payment-request events.
- `Payments`
  Owns payment state, payment event stream, provider integration boundary, and dispatching order execution after successful payment.
- `Tasks`
  Owns `TaskItem`, `TaskComment`, task lifecycle, and comments.

## Data Contracts to Change Before Extraction

- Replace user responses that embed orders with either:
  - a separate `GET /users/{id}` plus `GET /orders?userId=...`, or
  - a BFF-composed response.
- Replace direct entity relationships with IDs plus service contracts.
- Standardize event payloads with versioned schemas.
- Define RabbitMQ routing keys and message envelopes for `orders.*`, `payments.*`, and `tasks.*`.
- Model payment and order execution as idempotent consumers because delivery is at-least-once.

## Technical Decisions I'd Recommend

- Keep Keycloak as the single authentication authority.
- Use REST first between services unless you have clear low-latency internal RPC needs; add async events for state propagation.
- Use RabbitMQ as the broker for domain events and integration events.
- Use the outbox pattern before introducing a broker into production traffic.
- Use event sourcing for the `Payment Service`; treat the initial implementation as a stub with a real event stream and replayable projections.
- Use per-service DBs, not a shared database with multiple apps.
- Add OpenTelemetry from the start; distributed systems without tracing are harder to operate than monoliths.
- Use YARP if you want a .NET-native gateway quickly.

## Risks

- Biggest architectural risk: splitting too early while still depending on shared transactions and joins.
- Biggest integration risk: building RabbitMQ consumers and dispatchers without idempotency, dead-letter handling, and message versioning.
- Biggest operational risk: adding service count without observability, CI/CD maturity, and local development support.
- Biggest product risk: changing API shape too early, especially around user-and-order composite responses.

## When Not to Move Yet

Do not move yet if:

- the team is small and does not have CI/CD, observability, and on-call maturity,
- deployment pain is not currently a major problem,
- scaling needs are low and domain ownership is still fluid.

In that case, stop after Phase 1 or Phase 2 and keep a modular monolith. For this repo, that is a defensible interim state.

## Suggested Timeline

- Phase 0 to Phase 2: 5 to 10 weeks
- First extraction (`Tasks`): 3 to 5 weeks
- Second extraction (`Payment Service` stub + RabbitMQ flow): 3 to 5 weeks
- Third extraction (`Orders`): 3 to 5 weeks
- Final identity/user extraction and gateway hardening: 5 to 9 weeks

A realistic total is roughly 5 to 8 months for a careful migration, assuming one small team and no major product interruptions.
