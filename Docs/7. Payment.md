  Next changes should stay on the same workflow axis instead of jumping back to generic CRUD.

  1. Add a combined workflow endpoint.
     Right now the details page does two custom reads plus the base order read. That is workable, but not ideal.
     Recommended backend follow-up:

  - GET /api/orders/{id}/workflow
    Return:
  - order summary
  - payment details
  - timeline
  - payment events
    This reduces round trips and simplifies polling.

  2. Add explicit payment retry or re-request semantics.
     Once an order is PaymentFailed, the UI can show the reason, but the user still cannot do anything with it.
     Recommended backend:

  - POST /api/orders/{id}/retry-payment
    Recommended frontend:
  - show retry action only for failed orders
  - refetch workflow after mutation

  3. Add real execution outcome states.
     You currently stop at ExecutionDispatched.
     Recommended backend next states:

  - ExecutionStarted
  - ExecutionCompleted
  - ExecutionFailed
    Then expose them in the workflow/timeline read model.
    Frontend can then show actual end-to-end lifecycle instead of “dispatched and unknown after that”.

  4. Add payment/saga projection caching if this grows.
     Today handlers assemble the read model directly from saga + payment events, which is fine.
     If traffic grows, create a dedicated projection table for workflow details to avoid rebuilding on every request.
  5. Tighten order edit behavior.
     If not already done backend-side, lock down update paths so users cannot mutate workflow-owned fields like status directly through old endpoints.
     The frontend already moved away from free-form edits; the backend should enforce the same rule.
  6. Add frontend query encapsulation.
     The order details page currently knows too much about endpoint shapes.
     Recommended frontend refactor:

  - useOrderWorkflow(id, asUserId)
  - useRetryOrderPayment(id, asUserId)
    That keeps the page component thinner.

  My recommendation for the immediate next step:

  - implement GET /api/orders/{id}/workflow
  - then refactor frontend details page to one custom query
  - after that, add retry-payment

  That gives the cleanest next iteration with the least churn.
